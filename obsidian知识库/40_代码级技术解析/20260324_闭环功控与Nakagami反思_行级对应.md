## 2. 当前工程里有没有基于现有研究利用闭环功控

### 2.1 有，但分成两层，不是一个统一的完整闭环系统

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:654-665`
  - 这里定义了 `clpc_enabled / clpc_hour_of_day / clpc_users_by_type / clpc_capacity_by_type / clpc_pmin_frac_by_type`
  - 结论：工程里确实已经引入了“闭环功控”接口，但它是参数化近似，不是从真实网络状态实时回读的闭环。

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:1994-2044`
  - `_ue_ul_fpc_power_dbm()` 明确实现了蜂窝 UE 上行 `FPC` 公式
  - 结论：这里已经有基于现有蜂窝研究的分数功控，不是空白。

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:2066-2110`
  - `_calibrate_source_profile_and_power()` 中把 `cellular_ue_ul` 映射到 `_ue_ul_fpc_power_dbm()`，并在后面继续走 `_apply_clpc_power_control()`
  - 结论：当前代码已经形成“先按 UE UL FPC 标定，再对 AP/BS 类源应用 CLPC”的两层功控链。

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:3217-3282`
  - `_apply_clpc_power_control()` 明确写了：面向 `WiFi/AP/蜂窝基站类干扰源`，且说明“不是完整动态时序闭环”
  - 结论：现有实现是“时段 + 负载”的静态闭环近似，不应在论文里表述成完整网络级闭环控制。

### 2.2 反思

- 可以说“工程已利用现有研究中的 FPC / CLPC 思想”
- 不宜说“工程已经实现了完整真实闭环功控系统”
- 更稳妥的表述是：
  - UE 上行干扰采用研究启发的 `FPC` 标定
  - AP/BS 类干扰采用静态负载近似的 `CLPC` 功率调节

### 2.3 为什么这件事和公开测量数据要分开讲

- `compare_measured_cornercases.py:463-518`
  - `_align_bundle()` 在对齐测量点时还生成了 `station_position_m` 和 `hotspot_specs`
- `compare_measured_cornercases.py:582-610`
  - `_attach_aligned_environment()` 会把固定热点附着回场景

反思：

- 这说明“实测对照链”当前并不只是测量数据本身，还叠加了代理基站和经验热点
- 所以即使功控逻辑在工程里是成立的，也不能直接说这些功控是被公开数据集真实验证过的

## 3. 当前工程里有没有把全部干扰源换成 Nakagami 衰落

### 3.1 没有做到“全部源、全部路径、全部层面都换成 Nakagami”

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:812-825`
  - 定义了：
    - `interference_fading_enabled`
    - `interference_aggregation_model`
    - `nakagami_m_default / by_type / los / nlos`
  - 结论：Nakagami 参数体系已经存在，但是否启用取决于聚合模型和配置。

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:2265-2271`
  - `_precompute_interference_power_mw_per_drone()` 中，只要 `interference_aggregation_model` 是 `nakagami` 相关值，就走 `_nakagami_aggregate_interference_mw()`
  - 结论：默认总干扰聚合确实可以走 Nakagami 路径。

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:2611-2679`
  - `_nakagami_m_for_source()` 和 `_nakagami_aggregate_interference_mw()` 真正实现了 `m` 的选取和总干扰的等效 Nakagami 聚合
  - 结论：这里是“总干扰功率统计建模”的 Nakagami，不是把每一个干扰源对象都永久改写成 Nakagami 信道对象。

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:2114-2118`
  - `_stable_exp_fading_gain()` 写的是可复现的 `Rayleigh` 小尺度衰落功率增益
  - 结论：在线性聚合路径里，源码仍保留 Rayleigh 机制，因此不能说“全部干扰源已经统一换成 Nakagami”。

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\UAV_GA.py:2726-2728`
  - `calculate_interference_power_mw()` 再次按 `interference_aggregation_model` 分流到 Nakagami 或线性路径
  - 结论：当前是“可切换聚合模型”，不是“唯一 Nakagami”。

### 3.2 主入口并没有显式把它改掉

- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\compare_random_ga_gan.py:20-32`
  - `default_user_params()` 没有显式覆盖 `interference_aggregation_model`
- `成果本身\代码工程\无人机通信测试评估技术研究_代码与设计\0315\compare_measured_cornercases.py:43-58`
  - `_base_user_params()` 也没有显式覆盖 `interference_aggregation_model`

结论：

- 当前主流程大概率继承 `UAV_GA.py:817` 的默认值 `nakagami`
- 但这仍然不等于“全部干扰源已经统一换成 Nakagami 衰落”
- 更准确的说法是：
  - “当前总干扰聚合默认采用 Nakagami-m 统计近似”
  - “代码仍保留非 Nakagami 的线性 / Rayleigh 分支”

## 4. 应不应该进一步把全部干扰源都换成 Nakagami 衰落

### 4.1 不建议直接把原工程主线改成“全量 Nakagami 唯一路径”

原因 1：

- `compare_measured_cornercases.py:463-518`、`582-610`
  - 实测对照链本身已经叠加了代理基站和经验热点
- 反思：
  - 如果再把全部干扰都强行统一成 Nakagami，会把“数据语义偏离”和“统计信道假设偏离”叠加在一起
  - 这样主实验会更难解释

原因 2：

- `UAV_GA.py:2611-2679`
  - 当前 Nakagami 实现是“总干扰等效统计聚合”
- 反思：
  - 这更适合作为宏观随机干扰统计近似
  - 不等于所有具体干扰源都应该被逐个改写成 Nakagami 小尺度模型

原因 3：

- `UAV_GA.py:3217-3282`
  - AP/BS 类源已经有 CLPC 近似
- 反思：
  - 如果再把所有源统一换成 Nakagami，而没有按源类型校准 `m`
  - 可能会掩盖“功控变化”和“源类型差异”本身对结果的影响

### 4.2 更合理的做法

- 保留当前原工程不动
- 把 Nakagami 当成：
  - 一个可切换的统计聚合模型
  - 一个敏感性分析 / 消融实验选项
  - 而不是主线唯一信道

### 4.3 如果论文要写，建议这样表述

- 主线实验：
  - 不强调“所有干扰源统一采用 Nakagami”
  - 强调“总干扰聚合可采用 Nakagami-m 统计近似”

- 附加消融：
  - `Nakagami aggregate` vs `linear + optional Rayleigh`
  - 比较尾部风险、最差区域稳定性、GA/GAN 搜索结果是否变化

## 5. 结论（适合论文和汇报）

1. 当前工程已经引入了研究启发的功控机制
   - `UE UL -> FPC`
   - `AP/BS 类干扰源 -> 静态负载近似 CLPC`
   - 但不是完整真实闭环系统

2. 当前工程没有做到“全部干扰源统一替换成 Nakagami 衰落”
   - 现状更准确地说是：
     - 默认总干扰聚合倾向 Nakagami
     - 仍保留线性 / Rayleigh 路径

3. 不建议直接把原工程主线硬改成“全量 Nakagami”
   - 尤其在公开测量数据对照链里，会进一步放大建模假设

4. 更稳妥的做法
   - 原工程保持不动
   - Nakagami 作为附加统计敏感性分析
   - 测量数据主线应优先保持“测量场”语义清晰
