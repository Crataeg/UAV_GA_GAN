# 工程技术细节通俗解释（含公式推导与研究依据）

这份文档的目标很简单：把现在这套工程讲明白，让你既能写论文，也能给别人讲清楚为什么这样建模、这样对比。

## 1. 先用一句话说清楚这套工程在做什么

可以用一句话概括：

> 我们先在功率域里计算“有用信号多强、干扰和噪声多大”，得到 `SINR`，再由 `SINR` 推导出其余通信性能指标，最后比较不同算法能不能更稳定地找到更差的通信场景。

所以真正的核心不是“打分公式”，而是：

- 功率怎么来
- 干扰怎么聚合
- `SINR` 怎么算
- `SINR` 怎么推导到 `outage / rate / throughput / EE`

## 2. 为什么这套工程强调 SINR，而不是强调打分

因为 `SINR` 是最有物理意义的中间量。

如果把通信链路想象成“说话声”和“背景噪声”的关系：

- 有用信号越强
- 干扰和噪声越弱

我们越容易正确接收到信息。

`SINR` 衡量的就是这件事。

因此，最稳妥的论文主线应该是：

- `SINR` 是核心指标
- `outage / rate / throughput / EE` 是下游指标
- `avg_total_deg` 这类打分是优化器使用的工程映射层

## 3. 整体结构：三条主线

当前工程可以拆成三条线：

1. `城市与干扰源建模线`
2. `SINR-first 通信性能评估线`
3. `GA / GAN / Random 比较与 measured-field 对照线`

把这三条线接起来，就是整篇论文的方法部分。

## 4. 城市和干扰源到底是怎么建的

## 4.1 城市不是“真实地图复刻”，而是“有依据的统计城市”

这是一个必须讲清楚的边界。

当前主线默认不是逐楼复刻某个真实城市，而是用统计参数生成“有代表性的城市环境”。

对应代码：

- `UAV_GA.py:674`
- `UAV_GA.py:1074`

这里的城市参数是：

- `α`
  - 建筑覆盖率
- `β`
  - 建筑密度
- `γ`
  - 建筑高度分布尺度

这些参数决定：

- 建筑数量
- 建筑平面覆盖
- 建筑高度分布

### 4.1.1 建筑高度为什么用 Rayleigh

代码里是：

`height ~ Rayleigh(γ)`

对应：

- `UAV_GA.py:1098`

通俗解释：

- 楼高不会都一样
- 中低楼更常见
- 很高的楼少一些

Rayleigh 分布正好能表达这种“多数一般、少数很高”的统计结构。

### 4.1.2 建筑平面大小怎么来

代码里先根据：

- 城市总面积
- 覆盖率 `α`
- 建筑数量 `β`

反推平均单栋建筑该占多大面积，再加入随机扰动。

这样做的好处是：

- 不会出现“楼特别少但每栋占半个城市”
- 也不会出现“楼特别多但每栋都小得不合理”

### 4.1.3 什么是语义层

代码还生成了：

- `street`
- `sidewalk`
- `ground`
- `roof`
- `indoor`
- `pole`

对应：

- `UAV_GA.py:1524`

这些语义层不是装饰，而是用来约束：

- 干扰源部署位置
- UAV 位置投影

通俗理解：

- WiFi 更像在室内
- 蜂窝站更像在楼顶或杆塔
- 地面源更像在道路或地面

## 5. 从功率到 SINR：整套工程的核心公式链

这一段是全文最重要的技术主线。

对应核心代码：

- `UAV_GA.py:2846`

## 5.1 第一步：计算有用接收功率 `P_rx`

统一模板可以写成：

`P_rx[dBm] = P_tx[dBm] - PL[dB] - extra_loss[dB] + 10log10(fading) + 10log10(activity)`

对应：

- `UAV_GA.py:2564`

通俗理解：

- 发射功率越大，接收越强
- 路径损耗越大，接收越弱
- 衰落会让瞬时接收起伏
- 活动因子会把峰值影响折算成平均影响

## 5.2 第二步：计算聚合干扰 `I_y`

干扰不是只看一个源，而是所有干扰源在 UAV 位置上的叠加。

抽象写法：

`I_y = Σ I_i`

代码里有两种口径：

1. 线性叠加
2. `Nakagami-m` 聚合近似

对应：

- `UAV_GA.py:2657`

每个 `I_i` 会受到这些因素影响：

- 源功率
- 路径损耗
- 频谱耦合
- 衰落
- 活动因子

## 5.3 第三步：计算噪声 `N_y`

噪声按常见通信口径：

`N[dBm] = N0[dBm/Hz] + 10log10(B) + NF`

这里：

- `N0` 是热噪声谱密度
- `B` 是带宽
- `NF` 是噪声系数

通俗解释：

- 带宽越宽，总噪声越大
- 接收机越差，等效噪声越大

## 5.4 第四步：把干扰和噪声合成输入功率 `P_IN`

先在线性域相加：

`P_IN[mW] = I_y[mW] + N_y[mW]`

再转回 dBm：

`P_IN[dBm] = 10log10(P_IN[mW])`

## 5.5 第五步：定义功率裕量 `M_db`

代码里的关键中间量：

`M_db = P_rx[dBm] - P_IN[dBm]`

它其实已经等价于 `SINR` 的 dB 表示。

因为：

`SINR = P_rx[mW] / P_IN[mW]`

取 10 对数后：

`10log10(SINR) = P_rx[dBm] - P_IN[dBm] = M_db`

所以直接得到：

`SINR = 10^(M_db / 10)`

这一步很关键，因为它把“功率差”变成了真正的通信性能中间量。

## 6. 从 SINR 推导到其他通信性能指标

## 6.1 Outage

最直接：

`Outage = 1{SINR_dB < threshold}`

对应：

- `kpi.py:56`

通俗解释：

- 如果 `SINR` 比系统接受门限还低，这条链路就进入不可接受状态

## 6.2 Rate

当前工程采用 Shannon 上界口径：

`R = B log2(1 + SINR)`

对应：

- `UAV_GA.py:2846`
- `evaluate.py:101`

通俗解释：

- `SINR` 越高，单位带宽能承载的信息越多
- 所以 `rate` 和 `SINR` 单调相关

## 6.3 BLER-adjusted Throughput

吞吐不是简单等于理论速率，而是：

`Throughput = R * (1 - BLER(SINR))`

对应：

- `kpi.py:99`

为什么要这么做？

- 即便理论速率很高
- 如果误块率高
- 真正有效交付的数据量也会下降

### 6.3.1 BLER 从哪里来

当前工程里 BLER 来自 LUT：

- `BLER_A.csv`
- `bler_mc.py`

所以这层是：

- 系统级抽象

不是：

- 完整无人机专用 PHY 基带仿真

## 6.4 Energy Efficiency

代码里近似写作：

`EE = R / P_total`

其中：

- `P_total ≈ P_propulsion + P_tx`

对应：

- `evaluate.py:101`

这一步体现了无人机和普通地面终端的差别：

- 对无人机来说，推进功率不能忽略

## 7. 打分机制到底是什么

工程里有：

- `comm_deg`
- `avg_total_deg`

它们不是错误，也不是没用。

它们的作用是：

- 帮助优化器搜索
- 把多因素压成一个工程目标

但它们不是物理主角。

### 7.1 通信劣化映射

代码里本质上是把 `M_db` 映射到 `[0,1]`：

`D_comm = 1 / (1 + exp((M_db - θ) / s))`

对应：

- `UAV_GA.py:2975`

意思是：

- `M_db` 很差，劣化接近 1
- `M_db` 很好，劣化接近 0

### 7.2 综合劣化

综合劣化再结合速度/能耗层：

`D_total = weighted(D_comm, D_speed)`

对应：

- `UAV_GA.py:3756`

因此，论文最稳妥的写法是：

- `D_total` 是工程汇总指标
- `SINR` 及其派生指标是通信性能主线

## 8. 闭环功控和 Nakagami 在工程里的角色

## 8.1 UE 上行 FPC

对应：

- `UAV_GA.py:2024`

作用：

- UE 发射功率不是随便设的
- 它和路径损耗、资源块等因素有关

## 8.2 AP/BS 的慢时标 CLPC

对应：

- `UAV_GA.py:3247`

通俗理解：

- 负载低时，不需要一直满功率
- 负载高时，功率可以上调

但它不是：

- 真实每时隙网络调度闭环

所以论文里应写成：

- `slow-timescale CLPC approximation`

## 8.3 Nakagami 聚合干扰

对应：

- `UAV_GA.py:2657`

它解决的是：

- 多个干扰源叠加后，不能只看一个简单均值
- 需要一个更合理的统计聚合形式

因此它是：

- 聚合层近似

而不是：

- 每一个单独干扰源都被强行重写成“真实 Nakagami 信道”

## 9. GA、GAN、Random 到底分别在比什么

## 9.1 GA

`GA` 是最坏场景搜索器。

它直接优化：

- 干扰源类型
- 干扰源位置
- 干扰源功率
- UAV 高度/速度/位置

目标是把通信状态推向更差。

所以 `GA` 是：

- worst-case optimizer

## 9.2 GAN

`GAN` 不是直接替代 `GA`，它的角色是：

- 学习 GA 找到的高风险样本分布
- 生成相邻高风险样本

所以它更像：

- 高风险分布生成器
- 边界场景补样器

当前结果下，最稳妥的写法是：

- `GA` 的最坏搜索能力更强
- `GAN` 有高风险分布学习价值

## 9.3 Random

Random 很重要，因为它是：

- 同一边界下的无信息基线

没有 Random，就很难证明“算法真的起作用”。

## 10. 为什么要做 AERPAW / Dryad 对照

因为只做 synthetic，别人会问：

- 你这个结果是不是只在你自己设定的模型里成立？

所以要有 measured 对照。

## 10.1 measured-map 这条线到底做了什么

对应：

- `compare_measured_map_search.py:116`
- `compare_measured_map_search.py:163`
- `compare_measured_map_search.py:387`
- `compare_measured_map_search.py:702`

流程是：

1. 读取 measured SINR / throughput 点
2. 对齐到局部平面
3. 插值成 measured risk field
4. 在这个 field 上比较 `GA / GAN / Random`

## 10.2 为什么不能把它写成“真实 BS/干扰源地图”

因为公开数据给的是：

- 测量点
- 测量值

不是：

- 完整真实基站几何
- 完整真实干扰源真值地图
- 完整真实建筑附着热点图

所以最稳的写法只能是：

- `measured communication field`
- `measured degradation field`

## 11. 目前结果应该怎么通俗解释

### 11.1 Synthetic 结果

如果 `GA` 的 `mean SINR` 比 `Random` 更低：

- 说明 `GA` 更会把系统推向差状态

如果 `GAN` 的 `mean SINR` 比 `Random` 更低，但没有 `GA` 低：

- 说明 `GAN` 有风险学习作用
- 但最坏搜索能力仍不如 `GA`

### 11.2 measured 结果

如果 `GA` 的 `hit_rate_15m` 比 `Random` 高很多：

- 说明它更会找到 measured poor region

这就是“算法有作用”的核心证据之一。
