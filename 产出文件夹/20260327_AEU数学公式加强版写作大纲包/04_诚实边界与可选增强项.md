# 诚实边界与可选增强项

## 1. 哪些内容可以完全按“当前工程事实”写

这些内容可以直接写，不会越界：

1. 统计城市建模：`alpha / beta / gamma`、Rayleigh 楼高、语义层约束
2. `UE UL FPC` 与 `AP/BS slow-timescale CLPC`
3. `Nakagami/Gamma` 聚合干扰近似
4. `SINR -> outage / rate / throughput / EE` 的统一评价链
5. `GA / GAN / Random` 的对比关系
6. `AERPAW / Dryad` 作为 measured communication field 的验证

## 2. 哪些内容必须降级表述

### 2.1 `D_total`

必须写成：

- engineering objective
- optimization-oriented aggregate score

不要写成：

- physical law
- universal communication performance metric

### 2.2 `measured-map`

必须写成：

- measured communication field
- field-driven poor-region validation

不要写成：

- true BS map
- explicit interferer truth reconstruction
- real city full-network digital twin

### 2.3 `GAN`

必须写成：

- high-risk distribution learner
- generative augmenter around GA-discovered samples

不要写成：

- better worst-case optimizer than GA

## 3. 哪些增强项可以加，但必须标为扩展分析

### 3.1 `CVaR` 或 tail-risk 目标

可以在扩展版中写：

`maximize J_tail(x) = lambda_1 * E[D_comm(x)] + lambda_2 * CVaR_q(D_comm(x)) - lambda_3 * EE(x)`

但当前代码并没有把 `CVaR` 作为主目标求解，所以正文里只能写成：

- optional extension
- future enhancement

### 3.2 短包 finite blocklength 公式

AEU 很接受这一类公式，但当前工程主线仍是 `BLER LUT`。

因此：

- 可以在 related work 或扩展讨论里提及 short-packet direction
- 不要把 finite blocklength closed-form 写成当前工程已经实现的主结果

### 3.3 Meta-distribution 或更强随机几何结论

当前工程没有把 PPP 随机场景推导成完整随机几何 closed-form 网络级表达。

因此：

- 可以做附录式拓展
- 不能冒充当前实验主结论

## 4. 投稿时最危险的三种写法

### 危险写法 1

“本文提出了一种新的 GAN 最坏工况搜索算法，并证明其优于 GA。”

问题：

- 当前结果并不支持

### 危险写法 2

“本文基于 AERPAW / Dryad 重建了真实城市蜂窝网络和干扰源地图。”

问题：

- 数据语义不支持

### 危险写法 3

“本文用综合劣化指标严格刻画通信系统性能。”

问题：

- 审稿人会问为什么不用 outage / throughput / EE 这些标准通信指标

## 5. 最稳的主文写法

### 5.1 主文

- 写当前代码完全支撑的数学链
- 主打闭式 outage + measured-field validation
- `GA` 作为主搜索器，`GAN` 作为补样器

### 5.2 附录或扩展讨论

- `CVaR`
- finite blocklength
- 更强的平均速率积分分析
- 更细的 source-type sensitivity

## 6. 一句话总原则

让论文看起来更数学，不等于让论文离工程更远。

最好的做法是：

- 用当前工程真实存在的对象和变量，推导出更像通信系统论文的公式
- 而不是补一堆与代码脱节、与结果脱节的漂亮公式
